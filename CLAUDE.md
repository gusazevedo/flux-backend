# Flux Backend

## Project Overview
Personal finance API. Authenticated users can create income and outcome transactions,
list recent transactions, and view a dashboard with spending breakdowns by category.

---

## Tech Stack
- **Runtime:** Node.js + TypeScript
- **Framework:** Fastify
- **ORM:** Drizzle ORM
- **Database:** Supabase (PostgreSQL)
- **Auth:** Supabase Auth — JWT verification handled via Fastify plugin

---

## Architecture Rules (SOLID applied to this project)

### Single Responsibility
- Route handlers do exactly three things: parse the request, call a service method, return the response. Nothing else.
- Services contain all business logic and validation rules. They do not know about HTTP or Drizzle internals.
- Repositories contain all database queries. They do not contain business logic.
- If a function is doing two distinct things, split it.

### Open/Closed
- Add new behavior by creating new services, repositories, or plugins — not by modifying existing ones.
- Extend Fastify via plugins. Never modify `app.ts` to add route-level logic.

### Interface Segregation
- Each repository has its own interface (e.g., `ITransactionRepository`). Interfaces are small and specific to their domain.
- Services depend on the interface, not the concrete Drizzle class. This keeps layers decoupled.

### Dependency Inversion
- Services receive their repository via constructor injection — they never instantiate a repository directly.
- This makes services testable in isolation and swappable without touching business logic.

Example of the correct pattern:
```typescript
// interface
interface ITransactionRepository {
  findAllByUser(userId: string): Promise<Transaction[]>
  create(data: CreateTransactionDTO): Promise<Transaction>
}

// service depends on the interface
class TransactionService {
  constructor(private repo: ITransactionRepository) {}

  async listTransactions(userId: string) {
    return this.repo.findAllByUser(userId)
  }
}

// route handler delegates to service — no logic here
fastify.get('/transactions', async (request, reply) => {
  const transactions = await transactionService.listTransactions(request.user.id)
  return reply.send(transactions)
})
```

## Clean Code Rules

- **Names must reveal intent.** `getUserById` not `get`, `calculateOutcomeByCategory` not `calc`.
- **Functions do one thing.** If you need "and" to describe what a function does, split it.
- **No magic numbers or strings.** Extract to named constants.
- **No comments that explain what the code does** — write code that is self-explanatory. Only comment when the *why* is non-obvious (a workaround, a hidden constraint, a business rule not visible in the code).
- **Keep functions short.** If a function doesn't fit on one screen, it's doing too much.
- **Return early.** Avoid deep nesting by guarding at the top of a function and returning early on invalid conditions.
- **DTOs for data transfer.** Use explicit types for what goes in and comes out of each layer — never pass raw request bodies directly into services or repositories.

---

## Error Handling

- Validate all input at the route layer using Fastify's JSON Schema (`schema.ts` per route).
- Services throw domain errors with clear messages (e.g., `TransactionNotFoundError`).
- A centralized Fastify error handler in `plugins/` maps domain errors to HTTP responses.
- Never swallow errors silently. Never expose stack traces to the client.

---

## Conventions

- All routes require authentication unless explicitly marked as public in the spec.
- User ID always comes from the verified JWT (`request.user.id`) — never from the request body.
- All timestamps are stored and returned in UTC ISO 8601.
- Monetary amounts are stored as integers in cents to avoid floating point issues.
- Database migrations are generated by Drizzle Kit — never edit migration files by hand.
