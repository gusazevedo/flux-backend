# 02 — Transactions

## Overview
Authenticated users record **income** and **outcome** transactions and list their
recent transactions with filtering and cursor-based pagination. Every transaction
belongs to exactly one category owned by the same user.

Scope of this spec: `POST /transactions` and `GET /transactions`. Updating,
deleting, and fetching a single transaction are out of scope.

---

## Upstream dependency — `categories` table

Transactions reference a category through an enforced foreign key
(`category_id → categories.id`), so the `categories` **table** must exist first.

This spec therefore introduces the `categories` **table and its migration only**
— no category endpoints. Category management (create/list/update/delete) is
specced and built in `03-categories.md` on top of this already-existing table.
This keeps the `02 → 03` implementation order intact while satisfying the FK.

---

## Data model — `src/db/schema.ts`

Migrations are generated by Drizzle Kit — never hand-edited.

### Enum

```typescript
export const transactionType = pgEnum('transaction_type', [
  'income',
  'outcome',
]);
```

### `categories` (table only)

| column | type | rules |
|---|---|---|
| `id` | uuid PK | default random |
| `user_id` | uuid not null | owner, from JWT |
| `name` | text not null | category label |
| `created_at` | timestamptz not null | default now |

### `transactions`

| column | type | rules |
|---|---|---|
| `id` | uuid PK | default random |
| `user_id` | uuid not null | from JWT |
| `type` | `transaction_type` not null | `income` \| `outcome` |
| `amount` | integer not null | cents, **> 0** — direction comes from `type`, never a sign |
| `category_id` | uuid not null | **FK → `categories.id`, ON DELETE RESTRICT** |
| `description` | text nullable | optional free text |
| `occurred_at` | timestamptz not null | user-set; defaults to now if omitted |
| `created_at` | timestamptz not null | default now |

Add a composite index on `(user_id, occurred_at desc, id desc)` to back the
keyset pagination of `GET /transactions`.

**Foreign-key delete policy.** The `category_id` FK is `ON DELETE RESTRICT`: a
category that still has transactions cannot be deleted. This protects financial
records from accidental loss and avoids orphaned rows — `category_id` is `not
null`, so `SET NULL` is impossible, and `CASCADE` would silently delete a
user's transaction history. Spec `03-categories.md` builds category deletion on
top of this policy (e.g. requiring the category to be empty first). Because the
FK and its migration are generated **here**, this policy is fixed here.

## Layering & contracts

Follow the project architecture: thin route → service (business rules) →
repository (queries). The service depends on an **interface**, injected via the
constructor — it never instantiates a repository directly.

### DTOs

```typescript
interface CreateTransactionDTO {
  userId: string;
  type: 'income' | 'outcome';
  amount: number;        // positive integer, cents
  categoryId: string;
  description?: string;
  occurredAt?: string;   // ISO 8601; defaults to now
}

interface ListTransactionsQuery {
  userId: string;
  limit: number;
  cursor?: string;
  type?: 'income' | 'outcome';
  categoryId?: string;
  from?: string;         // ISO 8601, filters occurred_at >= from
  to?: string;           // ISO 8601, filters occurred_at <= to
}
```

### Domain entity — `src/interfaces/transaction.ts`

The repository returns a **camelCase domain entity**, never the raw Drizzle
row. This is the single shape every layer above the repository speaks.

```typescript
interface Transaction {
  id: string;
  userId: string;        // used by services for scoping; stripped at the HTTP edge
  type: 'income' | 'outcome';
  amount: number;        // positive integer, cents
  categoryId: string;
  description: string | null;
  occurredAt: string;    // ISO 8601 UTC
  createdAt: string;     // ISO 8601 UTC
}
```

**Mapping rule.** The repository is the *only* place that knows about
snake_case DB columns. It maps each row (`user_id → userId`, `occurred_at →
occurredAt`, `Date` → ISO 8601 string, …) to `Transaction` before returning.
Services and routes never see snake_case keys or `Date` objects.

### Interface — `src/interfaces/i-transaction-repository.ts`

```typescript
interface ITransactionRepository {
  create(data: CreateTransactionDTO): Promise<Transaction>;
  findManyByUser(query: ListTransactionsQuery): Promise<Transaction[]>;
}
```

`TransactionService` receives an `ITransactionRepository` via its constructor.
To enforce category ownership (below) it also depends on a category lookup
(e.g. `ICategoryRepository.existsForUser(categoryId, userId)`), injected the
same way.

### Wiring & composition

The **route plugin is the composition root**. Concrete repositories are
constructed with the shared Drizzle client (`src/db/client.ts`) and injected
into the service there. `app.ts` only registers the route plugin — it gains no
route-level logic, and the service never imports `db` directly, so Dependency
Inversion stays intact and the service remains unit-testable with fakes.

```typescript
// src/routes/transactions/index.ts
import { db } from '../../db/client';

export async function transactionRoutes(fastify: FastifyInstance) {
  const service = new TransactionService(
    new TransactionRepository(db),
    new CategoryRepository(db),
  );

  fastify.post('/transactions', { preHandler: fastify.authenticate, schema },
    (request, reply) => createHandler(service, request, reply));
  fastify.get('/transactions', { preHandler: fastify.authenticate, schema },
    (request, reply) => listHandler(service, request, reply));
}
```

Register it in `app.ts` alongside the existing route plugins:
`app.register(transactionRoutes)`. Unit tests construct the service directly
(`new TransactionService(fakeRepo, fakeCategoryRepo)`) — the wiring above does
not get in the way.

## `POST /transactions`

Protected. Creates a transaction for the authenticated user.

`user_id` is always taken from `request.user.id` — never from the body.

### Request body (TypeBox — `src/routes/transactions/schema.ts`)

| field | type | rules |
|---|---|---|
| `type` | enum | `income` \| `outcome` — required |
| `amount` | integer | `minimum: 1` (cents) — required |
| `categoryId` | string | uuid format — required |
| `description` | string | optional, `maxLength: 255` |
| `occurredAt` | string | optional, ISO 8601 date-time |

```typescript
const createTransactionBody = Type.Object({
  type: Type.Union([Type.Literal('income'), Type.Literal('outcome')]),
  amount: Type.Integer({ minimum: 1 }),
  categoryId: Type.String({ format: 'uuid' }),
  description: Type.Optional(Type.String({ maxLength: 255 })),
  occurredAt: Type.Optional(Type.String({ format: 'date-time' })),
});
```

### Validation split
- **Route layer (TypeBox):** shape and format — enum membership, positive integer
  `amount`, uuid `categoryId`, ISO `occurredAt`, `description` length.
- **Service layer (business rules):**
  - `categoryId` must exist **and belong to the authenticated user** — otherwise
    `404` (`CategoryNotFoundError`). Ownership is checked, not just existence, so
    a user can never attach another user's category.
  - `occurredAt` must not be in the future → `400` if it is.
  - `occurredAt` defaults to the current UTC time when omitted.
  - `description`: an empty string (`""`) is normalized to `null`.

### Response `201`

```json
{
  "id": "uuid",
  "type": "outcome",
  "amount": 1299,
  "categoryId": "uuid",
  "description": "Groceries",
  "occurredAt": "2026-05-31T14:00:00.000Z",
  "createdAt": "2026-05-31T14:02:11.000Z"
}
```

`userId` is not returned — it is implied by the authenticated caller.
`description` is `null` when omitted.

### Response schema (TypeBox — `src/routes/transactions/schema.ts`)

Declared as a Fastify **response** schema so serialization strips `userId`
automatically — only the fields below are ever sent. Shared by both routes.

```typescript
const transactionResponse = Type.Object({
  id: Type.String({ format: 'uuid' }),
  type: Type.Union([Type.Literal('income'), Type.Literal('outcome')]),
  amount: Type.Integer(),
  categoryId: Type.String({ format: 'uuid' }),
  description: Type.Union([Type.String(), Type.Null()]),
  occurredAt: Type.String({ format: 'date-time' }),
  createdAt: Type.String({ format: 'date-time' }),
});
// attach to the route: schema.response = { 201: transactionResponse }
```

### Errors
- `400` — body fails validation, or `occurredAt` is in the future.
- `401` — missing or invalid token.
- `404` — `categoryId` does not exist or is not owned by the user.

## `GET /transactions`

Protected. Returns the authenticated user's transactions, newest first, with
optional filters and cursor pagination. Only the caller's own rows are ever
returned.

### Query params (TypeBox)

| param | type | rules |
|---|---|---|
| `limit` | integer | optional, default `20`, `minimum: 1`, `maximum: 100` |
| `cursor` | string | optional, opaque base64 token |
| `type` | enum | optional, `income` \| `outcome` |
| `categoryId` | string | optional, uuid format |
| `from` | string | optional, ISO 8601 — `occurred_at >= from` |
| `to` | string | optional, ISO 8601 — `occurred_at <= to` |

### Pagination semantics (keyset / seek)
- Ordering: `occurred_at DESC, id DESC` (stable tiebreak on `id`).
- The `cursor` is an **opaque base64 string** encoding the `occurredAt` and `id`
  of the last row of the previous page (e.g. base64 of `"<occurredAt>|<id>"`).
- Next page predicate: `(occurred_at, id) < (cursorOccurredAt, cursorId)`.
- The handler fetches `limit + 1` rows: if the extra row exists, it is dropped
  from `data` and used to compute `nextCursor`; otherwise `nextCursor` is `null`.
- A malformed/undecodable cursor → `400`.

### Response `200`

```json
{
  "data": [
    {
      "id": "uuid",
      "type": "outcome",
      "amount": 1299,
      "categoryId": "uuid",
      "description": "Groceries",
      "occurredAt": "2026-05-31T14:00:00.000Z",
      "createdAt": "2026-05-31T14:02:11.000Z"
    }
  ],
  "nextCursor": "eyJvY2N1cnJlZEF0..."
}
```

`nextCursor` is `null` when there are no further pages.

### Response schema (TypeBox)

Wraps the shared `transactionResponse` from `POST /transactions`:

```typescript
const listTransactionsResponse = Type.Object({
  data: Type.Array(transactionResponse),
  nextCursor: Type.Union([Type.String(), Type.Null()]),
});
// attach to the route: schema.response = { 200: listTransactionsResponse }
```

### Errors
- `400` — invalid query params or malformed `cursor`.
- `401` — missing or invalid token.

## Business rules

- Direction is conveyed solely by `type`; `amount` is always a positive cent
  value (never signed).
- A transaction's category must be owned by the requesting user.
- `occurred_at` defaults to the current UTC time when omitted and may not be in
  the future.
- All reads and writes are scoped to `request.user.id`; a user never sees or
  affects another user's rows.
- Monetary amounts are integers in cents; all timestamps are UTC ISO 8601.

## Error handling

### Infrastructure introduced here

No centralized error handler exists yet (spec `01` did not add one), so this
spec introduces it — the same way it introduces the `categories` table to
satisfy its own FK.

**Domain errors** live in `src/errors/`. A base `DomainError` carries a
`statusCode` and a stable `code`; concrete errors extend it:

```typescript
// src/errors/domain-error.ts
abstract class DomainError extends Error {
  abstract readonly statusCode: number;
  abstract readonly code: string;
}

// src/errors/category-not-found.error.ts
class CategoryNotFoundError extends DomainError {
  readonly statusCode = 404;
  readonly code = 'CATEGORY_NOT_FOUND';
  constructor() { super('Category not found'); }
}
```

**Handler plugin** `src/plugins/error-handler.ts` registers `app.setErrorHandler`
and is registered in `app.ts`:

```typescript
app.setErrorHandler((error, request, reply) => {
  if (error instanceof DomainError) {
    return reply.status(error.statusCode).send({ message: error.message });
  }
  if (error.validation) {              // Fastify/TypeBox schema failures
    return reply.status(400).send({ message: 'Validation failed' });
  }
  request.log.error(error);            // logged internally, never returned
  return reply.status(500).send({ message: 'Internal Server Error' });
});
```

### Mapping

Services throw domain errors; the handler maps them to HTTP responses:

- Category not found / not owned → `404` (`CategoryNotFoundError`)
- Validation failures (schema or business rule) → `400`
- Missing or invalid token → `401` (from the auth plugin)

All error bodies use the same shape — `{ "message": string }` — matching the
auth plugin's `{ message: 'Unauthorized' }`. Never expose stack traces or raw
database errors to the client.

## Confirmed business rules

Confirmed with the product owner — no longer open:

1. **Future-dated `occurred_at`** — **rejected** with `400`.
2. **`description`** — `maxLength` of **255** chars; an empty string (`""`) is
   treated the same as omitted and stored/returned as `null`.
3. **List defaults** — `limit` default **20**, max **100**.
